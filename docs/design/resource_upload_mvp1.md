## 一、资源上传页面 · MVP 设计文档（精简版）

**设计日期**: 2025-11-18
**版本**: MVP v1
**目标**: 用 4–6 小时做出「能用且顺手」的上传页，先打通上传闭环，之后再迭代高级功能。

---

### 1. 范围与目标

**本次只做：**

* 单文件上传（点击选择，H5 端可选拖拽）
* OSS 直传（含简单进度条）
* 基础表单字段：标题、描述、分类、课程名称
* 前端必填校验
* 提交审核（调用后端 /resource）

**本次明确不做（后续 Phase 2+）：**

* 草稿自动保存 / 离开确认
* 智能分类推荐 / 自动填充分类
* 分片上传 / 断点续传
* 快捷键（Ctrl+Enter 提交等）
* 多文件/批量上传
* 积分自定义/资源定价相关设置

---

### 2. 页面入口 & 路径

* 路径：`/pages/resource/upload`
* 入口：

  * PC：资源广场页右上角「上传资源」按钮
  * 移动端：资源广场 FAB「+ 上传」

---

### 3. 页面结构（MVP 版）

#### 3.1 PC 端布局（居中卡片）

┌────────────────────────────┐
│  ← 返回      上传资源              │  顶部栏（无提交按钮）
├────────────────────────────┤

│────────────────────────────│
│  [表单区域：标题、描述、分类、课程] │  ② 资源信息
│────────────────────────────│
│  [上传须知（简单文字说明）]        │  ③ 上传须知（精简）
│────────────────────────────│
│              [提交审核] [取消]     │  ④ 操作按钮（底部）

* 外层容器 `max-width: 900px; margin: 40px auto; border-radius: 16px; box-shadow`，风格与详情页保持一致。

#### 3.2 移动端布局（全宽纵向）

* 同样顺序：上传区域 → 表单 → 上传须知 → 底部固定提交按钮。

### 4. 功能模块

#### 4.1 文件上传区域（MVP）

**交互：**

* H5：

  * 支持点击「选择文件」按钮
  * 可选：支持拖拽文件到上传区域（如果方便就加，不行就先只做点击）
* 小程序：

  * 只支持点击上传（`uni.chooseMessageFile` / `wx.chooseMessageFile`）

**状态三态：**

1. 未选择文件：

   * 显示图标 + 文案：

     > 拖拽文件到此处或点击上传（H5）
     > 点击上传文件（小程序）
   * 支持格式、最大 50MB 提示一行小字

2. 上传中：

   * 显示文件名 + 进度条
   * 百分比文本：`正在上传… 37%`

3. 上传完成：

   * 显示文件名、大小
   * 一个「重新选择」按钮（重新走上传流程）

**验证规则（前端）：**

| 项   | 规则                                               |
| --- | ------------------------------------------------ |
| 格式  | 允许：pdf, doc, docx, ppt, pptx, xls, xlsx, txt, md |
| 大小  | ≤ 50MB                                           |
| 文件名 | 建议 ≤ 100 字符（超过只截断显示，不拦截上传）                       |

验证失败统一用 `uni.showToast` + 上传区域下红字提示即可。

**上传流程（MVP）：**

```text
选择文件
  ↓（前端大小/格式验证）
调用 GET /resource/upload/signature?fileName=xxx
  ↓
构造 FormData，POST 到 OSS host
  ↓（onprogress 更新进度条）
成功 → 保存 fileUrl, fileSize, fileType
  ↓
允许提交表单
```

---

#### 4.2 表单字段（MVP）

**字段列表：**

| 字段          | 必填 | 类型   | 校验规则       | 说明       |
| ----------- | -- | ---- | ---------- | -------- |
| title       | ✅  | 单行文本 | 1–100 字    | 资源标题     |
| description | ✅  | 多行文本 | 10–500 字   | 资源描述     |
| category    | ✅  | 下拉选择 | 必选         | 资源分类     |
| courseName  | ❌  | 单行文本 | 0 或 1–50 字 | 课程名称（选填） |

**分类选项（简化版）：**

```ts
const categories = [
  { label: '课件', value: '课件' },
  { label: '试题', value: '试题' },
  { label: '笔记', value: '笔记' },
  { label: '教材', value: '教材' },
  { label: '实验报告', value: '实验报告' },
  { label: '其他', value: '其他' }
]
```

> MVP 里不做 “智能推荐分类”，只是普通 picker。

**校验触发：**

* `input` 时只做字数提示（灰字）
* `blur` 或提交前：做必填校验 → 字段下红字提示

示例错误文案：

* 标题：`请输入 1-100 字的标题`
* 描述：`描述需在 10-500 字之间`
* 分类：`请选择资源分类`

---

#### 4.3 上传须知（精简版）

只保留 2–3 行文本，靠近表单底部：

* 支持格式：PDF、Office 文档等（与前端校验一致）
* 单个文件最大 50MB
* 提交后将进入审核流程，**审核通过可获得 10 积分奖励**

不做复杂折叠、不做说明卡片组件。

---

### 5. 提交审核流程（MVP）

```text
点击「提交审核」
  ↓
1）检查：是否已有上传成功的 fileUrl
    否 → 提示「请先上传文件」
  ↓
2）前端校验表单字段
    有错误 → 高亮第一个错误字段，滚到那里
  ↓
3）调用 POST /resource
  ↓
4）成功：
    - Toast: "提交成功，待审核"
    - 跳转：返回资源广场 / 我的上传列表（二选一）
```

> MVP 不做“二次确认弹窗”“继续上传”之类，多一步多一点开发量。

## 二、积分规则统一规范（简短版）

这部分是整个“资源生态”的统一规则，和上传页、列表页、详情页、下载接口都要保持一致。

---

### 1. 积分系统总原则

* 积分是一个**简单的胡萝卜系统**，不要搞成金融系统。
* 对用户来说，需记住三句话：

  1. **上传资源，通过审核 +10 积分**
  2. **下载任意资源 -5 积分**
  3. **同一资源重复下载只扣一次**

---

### 2. 获取积分规则

1. **上传资源（审核通过）**

   * 行为：用户上传资源 → 管理员审核通过
   * 奖励：+10 积分 / 资源
   * 说明：审核拒绝不加分；重复上传不同资源可多次加分。

2. **其他获取方式**

   * 预留：签到、活动奖励等，后续再扩展。
   * 与本次上传页无直接关系，可在另一份文档定义。

---

### 3. 消耗积分规则

1. **下载资源**

   * 消耗：每种资源统一 **固定 -5 积分**
   * 统一价：前端不展示“下载价字段”，只展示文案：

     > 下载将消耗 5 积分
   * 去重规则：

     * 同一用户对同一资源，**第一次下载扣 5 分**，之后再次下载不再扣分（但记录下载次数）。
   * 自己资源：

     * 资源作者下载自己上传的资源 **不扣积分**（建议采用这个用户体验；若业务上不想特殊处理，可以显式写“不做特殊处理”）。

2. **其他消耗**

   * 暂无，后续如置顶资源、推广等再扩展。

---

### 4. 与“资源定价”的关系

* 当前阶段：**不支持单个资源自定义价格**。

  * 即：不在上传页显示“下载积分/score”字段；
  * 后端资源表中如果已经有 `score` 字段，可统一写死为 5，仅用于兼容。

* 若未来要做差异定价：

  * 建议在**管理员后台**编辑资源价格；
  * 前端用户上传页仍然不出现积分价格配置；
  * 列表/详情展示「需要 X 积分」，下载接口按照资源实际价格扣分。

---

### 5. 下载接口行为（简版说明）

在 `/resource/{id}/download` 中统一实现：

1. 检查登录
2. 检查是否已下载过该资源：

   * 已下载：直接返回下载链接，不再扣分；
3. 未下载：

   * 检查积分 ≥ 5：

     * 否则返回错误码：`POINTS_NOT_ENOUGH`
   * 扣除 5 积分
   * 记录下载行为
   * 返回下载链接

前端只需根据错误码：

* 积分不足 → 弹出“积分不足，请先去上传资源/参与活动获得积分”
* 其他错误 → 常规错误提示

---

### 6. 上传页与积分的文案联动

上传页面里**唯一与积分相关的内容**：

* 上传须知里一句提示：

> 当前规则：每次上传资源审核通过可获得 10 积分，下载任意资源会消耗 5 积分。

不再出现任何“积分输入框”“资源价位”等内容，一下就把复杂度砍掉了。

---

* 上传页 MVP 的 scope 清晰、4–6 小时是有可能的；
* 资源积分体系是统一、简单、可解释的；
* 后续如果要做“豪华版”（草稿、智能分类、分片等），可以在此文档基础上追加 Phase 2/3，而不用推翻重来。
