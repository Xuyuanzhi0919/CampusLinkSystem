3.5 下载资源功能
下载流程
const handleDownload = async (resource) => {
  // 1. 检查登录状态
  if (!isLoggedIn) {
    showLoginPrompt()
    return
  }

  // 2. 检查积分是否足够
  if (userPoints < resource.requiredPoints) {
    uni.showModal({
      title: '积分不足',
      content: `下载需要 ${resource.requiredPoints} 积分，您当前积分：${userPoints}`,
      showCancel: false
    })
    return
  }

  // 3. 确认下载
  uni.showModal({
    title: '确认下载',
    content: `下载将扣除 ${resource.requiredPoints} 积分`,
    success: async (res) => {
      if (res.confirm) {
        // 4. 调用下载 API
        const result = await downloadResource(resource.id)
        
        // 5. 扣除积分
        userPoints -= resource.requiredPoints
        
        // 6. 跳转到文件 URL（或触发浏览器下载）
        window.open(result.downloadUrl)
        
        uni.showToast({ title: '下载成功', icon: 'success' })
      }
    }
  })
}

## 一、信息架构层面：再补几块关键入口

你现在的页面叫「资源广场」，本质是**发现 + 浏览**，但围绕“资源”其实还有三个高频视角没写死在结构里：

1. **「我的」相关**
   建议在顶部或右上角加入口（或者 Tab）：

   * 我的收藏
   * 我的上传
   * 下载历史
     这些可以单独页面，也可以是资源广场里的 Tab，但**入口要显式**，否则用户会以为“下载完资源就消失在宇宙了”。

2. **上传者视角的状态反馈**

   * 在列表中，如果是当前用户上传的资源：

     * 显示：`[我上传]` 小标签
     * 审核中：下载按钮改为「审核中」，禁用点击，避免误解
   * 这样用户能一眼知道：**“这个是我自己发的，还在审核/已通过/被拒绝”**

3. **管理员 / 审核员专用视图**
   你在需求里提了“审核员/管理员”，但页面结构目前全是面向普通用户的。
   建议补一句设计决策：

   * 普通用户：`status ≠ 1` 的资源默认不出现在广场
   * 审核员模式：

     * 有一个「待审核」入口（独立菜单或 Tab）
     * 列表默认按 `status=0` 拉，支持：

       * 批量通过/拒绝
       * 按科目/上传时间排序
         技术上可以复用 ResourceCard，但增加 `mode="admin"`，展示审核按钮。

## 二、积分经济规则：把“坑”先写清楚

现在规则是：

* 上传通过：+10
* 下载：-5

建议补几条“边界条件”，否则以后产品一变更就会吵：

1. **同一资源重复下载算几次扣分？**
   推荐：

   * 同一用户对同一资源：**第一次下载扣分，再次下载不扣**
     写进文档，后端也好实现（建一张下载记录表 check 存在）。

2. **作者下载自己的资源**

   * 建议：**不扣分**，否则体验很迷惑。

3. **下载失败如何处理**

   * 如果调用下载 API 扣完积分但 OSS 下载失败：

     * 后端应有：**事务 / 补偿机制**，要么只在成功时扣分，要么失败回滚
   * 文档里可以加一句：

     > 下载行为以后端确认成功为准，失败会自动回滚积分，不会“白扣”。

4. **反作弊思路（文档可以轻描淡写提一下）**

   * 同一文件重复上传：通过文件 hash 去重
   * 连续大量相似标题的低质量资源，可以给管理员后台一个「疑似刷分」标签

## 三、搜索 & 筛选：再加一点“贴心小招”

你现在的搜索/筛选已经够用了，但有几个非常简单、体验提升很大的点可以加进去：

1. **点击标签直接筛选**

   * 在资源卡片的 `🏷️ 标签` 上，加交互：

     * 点击某个标签 → 直接应用为筛选条件（`tag = xxx`），并更新筛选栏提示“已按标签：xxx 过滤”。

2. **自动与排序联动**

   * 有关键词搜索时：

     * 默认 `sortBy = relevance`（相关性排序，后端如果用 ES 就顺手搞一下）
   * 无关键词时：

     * 默认 `sortBy = time`（最新）

3. **筛选条件持久化再升级一点**

   * 现在你写的是 localStorage
   * 可以再加一句：**登录用户优先走后端保存个人偏好**（比如“偏好资源类型/学科”），多端同步会更舒服。

4. **搜索结果态文案**

   * “找到 X 条资源”那一栏，加两种情况：

     * 有结果：`为你找到 X 条相关资源`
     * 无结果：`没有找到相关资源，试试：减少筛选条件 / 换个关键词`
